# Android

# The Android Activity

Activities are one of the basic building blocks of an Android application.

Activities are the views (screens( for your app. They can contain one or more user interface components that the user can interact with. In many cases your app will contain multiple activities to handle the different screens used in your app.

Having multiple activities performing different tasks may give the impression that developing Android activities is a complicated undertaking.

However, Android simplifies this process by using specific callback methods that initiate code only when it is required. This system is known as the Android Activity Lifecycle

# The Android Intent

Most people don’t wander around each day aimlessly, without purpose.

Almost everything they do, (reading, eating, learning) has a purpose, or intent, behind it.

Android works in much the same way. Before an app can perform any action, it needs to know what the purpose, or intent, of that action is, so that it can perform that action correctly.

An Android intent is a messaging object you can use to request an action.

Intents are used when starting a new Activity.

Intents use a Bundle object, referred to as a bundle, to transfer data between activities as key/value pairs.

For more information see the documentation:

https://developer.android.com/reference/android/content/Intent (opens new window)

# Android Activity LifeCycle

There are six important Activity lifecycle methods:

  • onCreate()
  • onStart()
  • onResume()
  • onPause()
  • onStop()
  • onDestroy()

onCreate()

When an Activity first starts running, this method is called. onCreate() is executed only once during the lifetime.

onStart()

This method is called just before the Activity is rendered on screen.

onResume()

When the Activity is rendered on the screen this method is called. At this point, the Activity is in the active state. onResume() is also called when the Activity comes back to focus from the paused state.

onPause()

This method is called when the activity loses its focus and is only partially visible to the user. At this point the Activity is the paused state.

onStop()

This method is called when the Activity is in the active state then switches to the stopped state. This happens when the Home key is pressed: the Activity goes into the background and the Home Screen is visible. When this happens the Activity enters the stopped state and both the onPause() and the onStop() methods are executed.

When re-opening an app after pressing the Home key the lifecycle changes from the stopped state to the active state. This causes both the onStart() and the onResume() methods to be called.

onDestroy()

This method is called when an activity is closed (destroyed) by a user or by the Android OS.

# FullScreen Mode

Android offers three options to making your app full-screen:

  • Lean Back
  • Immersive
  • Immersive Sticky

All three approaches hide the system bars however your activity continues to receive all touch events. The difference between them is how the user can bring the system bars back into view.

Lean back mode is for full-screen experiences in which users won't be interacting heavily with the screen, such as while watching a video. When users want to bring back the system bars, they simply tap the screen anywhere.

Immersive mode is intended for apps in which the user will be heavily interacting with the screen. Examples are games, viewing images in a gallery, or reading paginated content, like a book or slides in a presentation. When users need to bring back the system bars, they swipe from any edge where a system bar is hidden. By requiring this more deliberate gesture, the user's engagement with your app won't be interrupted by accidental touches and swipes.

Sticky immersive mode: In the regular immersive mode, any time a user swipes from an edge, the system takes care of revealing the system bars—your app won't even be aware that the gesture occurred. So if the user might actually need to swipe from the edge of the screen as part of the primary app experience—such as when playing a game that requires lots of swiping or using a drawing app—you should instead enable the "sticky" immersive mode. While in sticky immersive mode, if the user swipes from an edge with a system bar, system bars appear but they're semi-transparent, and the touch gesture is passed to your app so it app can also respond to the gesture.

// region hide/show UI

    private fun hideSystemUI() {
        // Enables regular immersive mode.
        // For "lean back" mode, remove SYSTEM_UI_FLAG_IMMERSIVE.
        // Or for "sticky immersive," replace it with SYSTEM_UI_FLAG_IMMERSIVE_STICKY
        window.decorView.systemUiVisibility = (View.SYSTEM_UI_FLAG_IMMERSIVE
                // Set the content to appear under the system bars so that the
                // content doesn't resize when the system bars hide and show.
                or View.SYSTEM_UI_FLAG_LAYOUT_STABLE
                or View.SYSTEM_UI_FLAG_LAYOUT_HIDE_NAVIGATION
                or View.SYSTEM_UI_FLAG_LAYOUT_FULLSCREEN
                // Hide the nav bar and status bar
                or View.SYSTEM_UI_FLAG_HIDE_NAVIGATION
                or View.SYSTEM_UI_FLAG_FULLSCREEN)
    }

    // Shows the system bars by removing all the flags
    // except for the ones that make the content appear under the system bars.
    private fun showSystemUI() {
        window.decorView.systemUiVisibility = (
                View.SYSTEM_UI_FLAG_LAYOUT_HIDE_NAVIGATION
                or View.SYSTEM_UI_FLAG_LAYOUT_FULLSCREEN)
    }

// endregion

# Glide DiskCacheStrategy

There are several options for Glide v4 diskCacheStrategy:

  • diskCacheStrategy(DiskCacheStrategy.ALL)
  • diskCacheStrategy(DiskCacheStrategy.NONE)
  • diskCacheStrategy(DiskCacheStrategy.DATA)
  • diskCacheStrategy(DiskCacheStrategy.AUTOMATIC)
  • diskCacheStrategy(DiskCacheStrategy.RESOURCE)

diskCacheStrategy.NONE caches nothing

diskCacheStrategy.DATA caches only the original full-resolution image

diskCacheStrategy.RESOURCE caches the final image, after reducing the resolution

diskCacheStrategy.AUTOMATIC intelligently chooses a cache strategy based on the resource

diskCacheStrategy.ALL caches all versions of the image

# AsyncTask

Android AsyncTask is an abstract class provided by Android which allows you to perform asynchronous tasks in the background and prevent blocking in the UI thread.

Android applications run on a single thread when launched.

Blocking can occur with this single thread model when performing tasks that significant time to complete such as downloading content and accessing files. Blocking will cause the application to be non-responsive.

To help avoid this issue Android provides the AsyncTask class which can perform tasks in the background on a dedicated thread and then pass the results back to the UI thread.

The basic methods used in an android AsyncTask class are defined below:

  • doInBackground(): This method contains the code which needs to be executed in the background. This method can also send progress results multiple times back to the calling thread (UI) using the publishProgress() method. To notify that the background processing has completed we can simply use the return statement.
  • onPreExecute() : This method contains the code that is to be executed before the background processing starts.
  • onPostExecute() : This method is called after the doInBackground() method completes processing. The result from doInBackground is passed to this method.
  • onProgressUpdate() : This method receives progress updates from doInBackground() method, which is published using the publishProgress() method. This method can use this progress update data to update the calling (UI) thread.
AsyncTask<Params, Progress, Result>
1
  • Params, the type of the parameters sent to the task upon execution.
  • Progress, the type of the progress units published during the background computation.
  • Result, the type of the result of the background computation.

# Permissions

Prior to the introduction of Marshmallow (Android 6) the only step necessary for an app to request permission to access device functionality was to add the appropriate line into the application’s manifest file.

The user would then be prompted to approve the permission when the app was installed.

This still holds true in Android 6 and above except for permissions that are considered dangerous:

Permissions that are considered dangerous could have the potential to allow an app to violate the user’s privacy such as allowing access to the microphone, contacts list or external storage.

Apps based on Android 6 or above must now request dangerous permission approval from the user when the app launches in addition to requiring the permission request be put in the manifest file.

# File Storage

All Android devices have two file storage areas: internal storage (opens new window) and external storage (opens new window).

These names come from the early days of Android, when most devices offered built-in non-volatile memory (internal storage), plus a removable storage medium such as a micro SD card (external storage).

Many devices now divide the permanent storage space into separate "internal" and "external" partitions. So even without a removable storage medium, these two storage spaces always exist, and the API behavior is the same regardless of whether the external storage is removable.

Because the external storage might be removable, there are some differences between these two options:

# Internal storage:

It's always available. Files saved here are accessible by only your app. When the user uninstalls your app, the system removes all your app's files from internal storage. Internal storage is best when you want to be sure that neither the user nor other apps can access your files.

# External storage:

It's not always available, because the user can mount the external storage as USB storage and in some cases remove it from the device. It's world-readable, so files saved here may be read outside of your control. When the user uninstalls your app, the system does not remove your app's files from here unless your App forces this to happen. External storage is the best place for files that don't require access restrictions and for files that you want to share with other apps or allow the user to access with a computer.

# RecyclerView

Android has both a ListView class and a GridView class that can be used for displaying lists.

The newer and more advanced RecyclerView can be looked at as a combination of both a ListView and a GridView.

Additionally, the RecyclerView has features that allow for code separation so that you create individual components of the RecyclerView that are easier to maintain.

# Creating a RecyclerView:

There are 4 main steps when creating a RecyclerView:

  1. Declare a RecyclerView in your activity layout.
  2. Create a custom XML layout for your RecyclerView.
  3. Create a RecyclerView Adapter ( Create a view holder for view items, then connect the data source for the RecyclerView).
  4. Attach the adapter to your RecyclerView.

For more information see the documentation:

https://developer.android.com/reference/android/support/v7/widget/RecyclerView (opens new window)
